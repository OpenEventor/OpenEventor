Описать мониторинг сплитов




Монитор - это инструмент для realtime отслеживания отсечек passings.
При входе на страницу, происходит подключение к ServerSideEvent процессу.
Важно чтобы на бэкенде для одного мероприятия был один процесс (горутина) для публикации отсечек для фронтэнда, посколько сразу несколько судей на нескольких девайсах могут подключаться и мониторить появляющиеся отсечки.

Отсечка - это факт пересечения чипом какой либо контрольной точки на дистанции в какое-то время.
Например:
{
    id: "82974744-b335-4379-86f5-bc65e6f262c9", // ID отсечки
    card: "780EF410", // Номер карточки участника
    checkpoint: "LAP", // Название точки пересечения
    timestamp: 1772190954.45, // Точное время пересечения
    enabled: true, // Включена или отключена данная отсечка (например если ошибочкая - ее не удаляют, а отключают)
    source:"handtime app" // Кто создал эту отсечку
}

Во время мероприятия, отсечки приходят нонстоп: участники пересекают промежуточный или обычный финиш и оборудование (или ручной хронометраж) через API отправляет в приложение новые отсечки. Для теста нам необходимо будет создать отдельную утилиту, которая на указанный эндпоинт будет отправлять эти отсечки, чтобы симулировть настроящие мерояритие. 

И конечно нужно будет разработать универсальный модуль "Приёмник отсечек" для приема отсечек от различного оборудования/приложений через API. Будет единый интерфейс и несколько конфигурационных файлов, типа "sfr", "ostis2", "sportident" и тд. Разработчики смогут добавить свое оборудование. В приложении в этом модуле можно будет внести специфичные для системы электронной отметки настройки. Например "корректировка времени" для ostis2, или замена названий отсечек для SFR (будем менять 231 на "START" или как-то так).

Монитор отображает отсечки, группируя их в строки по участникам (через номер card) и сортируя по времени (самые свежие, появляются в конце своей строки и вся стока оказывается наверху), таким образом самые последние отсечки оказываются навреху справа;

Во фронтенде обязательно должна быть виртуализация, поскольку отсечек может быть 10-20 тысяч за мероприятие. Стоит использовать локальную базу данных (дай совет какую) и какую-то систему для виртуализации (тоже предложи варианты) и рендеринга блоков отсечек. Размер строки всегда будет фиксированный. Переносов отсечек на новую строку не должно быть.

Блок отсечки выглядит как небольшой прямоугольник со скругленными углами, с названием отсечки (крупно), ниже время прохождения и еще чуть ниже - мелким шрифтов - разница времени с прошлой enabled отсечкой (это можно вычислять локально). Тут важно не забывать про часовой пояс мероприятия (timezone будет храниться в settings, по умолчанию UTC). Фон отсечки - либо primary либо серенькая (в зависимости от enabled true/false). При правом клике - открываем контекстное меню, там можно поправить время или деактивировать или изменить название)

Между отсечками - маленькие вертикальные прямоугольники (высотой с блок отсечки, шириной 6 пикселей, с onhover цветом и боковыми марджинами в 2 пикселя) - при клике на который можно созадать между ними новую отсечку, которая будет сохранена в базе данных (обсудим позднее, пока просто сделать маленькие прямоугольники между отсечками и вначале и в конце, чтобы отсечку можно было создать в любом месте).

Блоки отсечек организованы в группы (по участнику - через card1/card2).
Группа в самом начале имеет блок "Участник" - там выводит bib-номер участника, мелким шрифтом имя-фамилия. Контекстный клик с дропдаун меню. 

Очень важный момент: в интерфейсе монитора должна быть кнопка Play/Pause.
Ее идея в том, чтобы приостанавливать рендеринг отсечек. При этом процесс получения отсечек с сервера продолжается ВСЕГДА. 
Например при добавлении новой отсечки вручную - ставим процесс рендера на паузу.

Нужно проработать логику: что мы формируем на бэкенде, что на фронтэнде. 
В будущем на фронтэнде нужно будет иметь возможность фильтровать этот список по дистанциям/группам и другим параметрам. Но фильтр давай оставим на потом. Просто держи в контексте тот факт что фильтр будет нужен. Возможно это повлияет на набор данных. Иногда будут происходить серьезные изменения, например участнику в другой вкладке сменили дистанию и это значит, нужно будет обновить его данные на фронте. Кароче получается что бэкенд всегда должен отправлять изменения которые могут повлиять на внешний вид. 

Так же важный момент будет - фильтр "Проблемные" - должен будет отображать только тех участников, у которых проблема с прохождением дистанции (нарушем порядок прохождения)
